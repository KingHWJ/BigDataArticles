#面试#数据仓库#成长

数据研发(数仓开发工程师）常常被戏称为SQL Boy/Girl，因为大部分的时间都是在使用SQL做ETL的工作。面试造火箭，入职拧螺丝，时间久了，成就感变的很低，经常会怀疑数据研发的价值体现在哪里呢？

本文将基于对数据研发工作的一些浅薄的理解，聊聊数据研发的能力模型，希望对你有所帮助。

技术能力

数据技术能力

关于大数据技术，其实浏览下招聘网站就可以了解市场上需要哪些大数据技术了。大数据发展到现在这个阶段，涌现的成熟技术不胜枚举，需要注意的是，这些技术只是工具，作为数据研发也没有必要都要学习一遍，当然能够有所了解是最好的了。

下面是我从Apache官网截的一张图，被框出来的大家应该都应该听说过：

我对常用的大数据技术栈归了一下类，当然没有囊括所有的技术，在这些技术中，对于Hive、Hadoop、Spark、Flink、Kakfa，我觉得作为数据研发，都应该去学习掌握的，因为在实际的开发过程中，多数情况下是会使用到的。另外，还需要掌握一门编程语言，常见的Java、Scala、Python，其实只要有Java基础，对于Scala和Python而言，应该很容易上手。

  1. 数据采集：如 Apache Flume、Apache Kafka 等技术用于采集、传输和聚合数据。
  2. 数据存储：如 Apache Hadoop HDFS、Apache Cassandra、MongoDB、HBase 等技术用于存储和管理大量数据。
  3. 数据处理：如 Apache Spark、Apache Flink 等技术用于处理和分析大规模数据。
  4. 数据分析：如 Apache Hive、Apache Impala、Clickhouse 等技术用于分析和挖掘数据。
  5. 编程语言：Java、Scala、Python

记住一句话，技术永远是工具，切不可陷入而不可自拔，我理解大数据的核心是数据，掌握了一堆技术，没有应用场景，其实意义不大，在实践中学，通过解决某个难题而应用技术，这才是重要的。当然，我不是说技术不重要，技术应该成为一个基础能力，这个基础能力能够让你快速学习并应用一门新技术，也就是所谓的融会贯通，才是我们应该追求的，只是浮于表面的学习新技术，只会浪费精力，真正的作用和效果并不明显。

数据模型能力

数据建模是数据研发工程师非常重要的一项能力，可以说是非常核心的能力。当然，建模比较抽象，不同的人也会有不同的理解，所以模型设计也千差万别，好的模型能够兼容多个取数场景，便于取数，差的数据模型，往往会导致数仓越来越乱，与数仓的理念背道而驰。

数据建模不是一蹴而就的，是在大量的需求中摸索出来的，是建立在对业务的理解基础之上的，当然，方法论也是非常重要，掌握了基本的方法论，再在实际的开发中加以应用，这样才能够真正理解其中的精髓。

工程能力

掌握编程语言和工具：数据开发工程师需要掌握至少一种编程语言（如Python、Java、Scala等）和相应的编程工具（如PyCharm、IntelliJ IDEA等），能够编写高质量的代码，并能够使用编程工具进行调试和优化。

其实，有时候数仓开发并不是仅仅写写sql就完事了，有时候是需要借助编程语言来实现一些复杂场景的数据处理的，比如会用到Python，Java结合SQL去处理一些数据。所以，具备良好的工程能力，对与日常的工作是非常重要的。

综合能力

前面提到了三方面的技术能力，分别数据处理、数据建模，基础的编码能力，这些都是是数据研发的基本技能，相信每个数据研发都应该有所掌握，接下来我再聊聊数据研发应该有的综合能力，也可以称之为软实力，这些能力只能在实际的工作中摸爬滚打中学习总结，只有经历了，躬身入局，才能够有所成长。

业务理解能力

  • 对支持业务/所属产品有深入的理解，能熟练地进行业务/产品需求分析，能较好地完成业务/产品需求。

  • 能对业务/产品具有独立沟通、完善业务需求的能力，并能够识别方案风险、很好地解决业务问题。

  • 能够在负责的业务/产品上有独立的见解，能够发现风险，能够提出合理的建议。

  • 对于定义问题、分析问题、解决问题的有较好的思维和方式。

很多研发会感觉到天天都在做需求，但是对于需求的了解也仅限于产品PRD上的需求背景介绍，比较点状，不够全局；建议研发需要找到业务或者产品负责的同事，主动去了解业务的规划、目标及策略，核心项目，透过这些目标、核心项目看清业务的运营思路，只有明白了这条线，才能确保主要的精力投入到重要工作中，不至于工作跑偏，貌似做了很多，却又感觉什么都没有做，陷入自我怀疑之中。

数据应用的场景也不再仅限于报表监控，数据与算法、产品的结合创造了多样的应用生态，个性化推荐、智能风控、用户画像，人群标签等等，这些场景已经深深地与业务绑在一起，这就要求我们的数据研发同学的思维不能仅仅停留在开发角色，研发需要逐步往理解熟悉业务，基于自身对全域数据理解的核心竞争力，与业务、技术协同，探索用数据帮助业务往更好的方向发展。

跨团队沟通协同

对于一个快速发展的企业来说，研发资源紧张的确是常态。因为要做的事情实在太多，在资源有限的情况下，很难一下子支持所有的业务需求，在这种情况下，区分重要紧急，排优先级是非常必要的。数据团队在处理某些需求时，不能仅仅从自身视角出发，还需要与业务进行充分的沟通，了解业务的需求和规划，给出合理的建议，确保双方的认知是一致的。

同时，数据团队还需要主动走出去，了解上下游的规划和动态，避免闭门造车。在跨团队边界方面，要尽可能做到高水平的协同，避免低水平的重复建设。因此双方团队如果有交叉还是就事论事讲清楚(这里常见的是BI、数据研发、算法之间的协同)，让专业的人做专业的事情，高效协同，拿到最好的结果，让业务赢是第一考虑的。

项目管理能力

项目管理的主要关注点是成本、时间和质量三个点。在日常需求处理中，我们需要多问几个为什么，首先我们要了解以下内容：

  • 业务背景是什么

  • 解决的业务或用户痛点是什么

  • 数据的使用者和期望达到的业务目标是什么

这些问题的答案会帮助你来判断绝大多数需求的优先级和重要程度。其次，我们需要确认所需数据的范围、更新频率、指标统计口径等问题，这些问题会帮助你确认开发所需的人日和预估交付时间。最后，我们需要与业务同学确认约定的交付日期、数据检验标准等，并保障自测结果符合预期，配置监控规则等。

在项目管理中，一个好的项目经理需要具备多种技能和素质，包括技术能力、沟通能力、协调能力、团队管理能力和项目管理能力等。在处理冲突和资源不足等问题时，需要有良好的解决问题的能力，能够合理地分配资源，保证项目的顺利进行。同时，需要有整体把控和协调整个项目相关的产品、设计、研发等团队的能力，确保项目按照预定的时间、质量和成本交付。

在数据开发中，了解业务需求和数据使用场景是非常重要的。数据开发工程师需要与业务团队进行充分的沟通和合作，理解业务的目标和需求，提供合理的建议和解决方案。在处理数据时，需要考虑到数据的质量、可靠性和安全性等方面的问题，并采用合适的数据处理技术和工具，确保数据的准确性和可用性。同时，数据开发工程师还需要掌握一定的数据分析技能，能够从数据中发现问题和机会，为业务提供有价值的洞察和决策支持。

个人影响力

  • 建立专业领域知识：数据开发工程师需要不断更新自己的技术知识和业务知识，了解行业趋势和最新技术，保持学习和进步的态度，成为团队内的技术专家，从而获得团队成员的信任和尊重。

  • 建立良好的沟通能力：数据开发工程师需要与团队内的其他成员进行有效的沟通，了解他们的需求和问题，积极解决问题，并与他们建立良好的合作关系。同时，还需要与其他部门的同事进行有效的沟通，建立良好的合作关系。

  • 主动参与团队活动：数据开发工程师可以主动参与团队的活动和讨论，积极为团队贡献自己的经验和知识，帮助团队成员解决问题，提高团队的整体水平。

  • 培养团队合作精神：数据开发工程师需要培养团队合作精神，与其他团队成员进行良好的协作和配合，共同完成团队的任务和目标。通过合作和协作，可以不断提高自己的团队影响力。

总之，数据开发工程师可以通过建立专业领域知识、良好的沟通能力、主动参与团队活动、培养团队合作精神和关注团队成员的发展等方面来打造自己的团队影响力，成为团队内的技术专家和领袖人物。

成长路径

大多数人的成长过程确实是按照点、线、面的路径发展的。

  • 在最初的阶段，我们负责的是一个个零散的需求开发交付，通过这个过程，我们可以熟悉技术的原理和特性、开发过程的规范、上下游的合作与沟通等。这个阶段的目标最重要的是要让我们能够被同事和团队信任，具备执行力，能够将任务有结果、有总结、有沉淀地完成。

  • 在第二阶段，我们可能会开始负责一块业务或一个产品，此时我们需要开始考虑如何运营，即如何更好地提高业务结果。我们需要与合作的PD、业务人员进行沟通，理解业务痛点，并思考如何用技术方案解决业务问题。这个阶段的目标或标志是能够主动规划、思考，并参与到业务的规划讨论中，成为业务的伙伴。

  • 在第三阶段，我们不再局限于某个业务或产品，而是具备了更丰富的技术积累和抽象能力，可以成为业务领域的专家，对业务进行诊断并提供端到端的解决方案。我们可以制定业务线的数据规划和落地方案，并联动上下游一起拿到结果。这个阶段的目标是成为业务和技术之间的桥梁，帮助业务成功。

总结

本文主要阐述了数据研发应该具备的哪些能力，当然成长不是毕其功于一役的事情，把事情踏踏实实做好，让自己有成长才是我们应该做的。最后还是数据研发还是要跳出岗位本身回归到个人，建议数据研发同学不要拘于工作本身，要沉淀和提升自身的可转移能力，能够理解洞察技术的原理和应用的深度广度，理解商业本身的逻辑，在面临瓶颈时可以换一个视角来看待问题，或许柳暗花明又一村。
